# 诉讼可视化Skill

## 功能概述

根据《诉讼精细化》办案手册第八节要求，运用Mermaid图表语法，将复杂的案件法律关系、时间脉络、争议焦点、要件分析、攻防对抗和证据链条转化为直观的可视化图表。按总控当前事项记录衔接，默认生成本地图片交付包；预览稿/工作稿须经图表出稿前审查和律师确认后，才制作正式图片交付稿。

## 依赖Skill

本Skill需要调用 **mermaid-diagrams** Skill 来生成Mermaid图表代码。

## 触发条件

- 【客户编号】+ "可视化"
- 【客户编号】+ "诉讼可视化"
- 【客户编号】+ "制作图表"
- 用户要求“输出案件可视化图表”“案件思维导图”“案件全套图表”“案件图谱”等类似表达时，默认生成全套案件图表，不只生成单张图。

## 第二部分：图表类型说明

本Skill支持7种基础图表类型，但交付逻辑以“用户最终查看是否清晰”为准，而不是固定套用某一种 Mermaid 语法。所有图表必须生成图片；Mermaid 源码只作为可编辑中间文件保存。

| 图表类型 | 首选展示形态 | 主要用途 |
|---------|-------------|---------|
| 法律关系图 | 分组关系图/主体网络图 | 展示当事人间的法律关系、权利义务流向 |
| 法律事实时间图 | 时间轴/分阶段时间轴/泳道时间轴 | 展示案件关键时间节点和事件脉络 |
| 金额流向图 | 资金流向图/分组流程图 | 展示款项支付、流转、占有、返还或去向不明 |
| 争点结构图 | 树状图/分层争点图 | 展示争议焦点的树形结构和状态标注 |
| 要件分析图 | 请求权-要件-事实-证据树状图 | 展示请求权→构成要件→证据的层级关系 |
| 攻防对抗图 | 左右对照泳道图/攻防矩阵图 | 左右对照展示我方主张与对方抗辩 |
| 证据链图谱 | 按证明对象分组的证据链图 | 按证明对象分组展示证据关系 |

> 📖 详细语法示例和标注规范参见 [references/六类Mermaid图表完整说明.md](六类Mermaid图表完整说明.md)

### 2.1 图片优先与可读性规则

1. 每一种法律图都必须导出图片，优先为 `.png`，必要时同时导出 `.svg`；不得把 Mermaid 代码当成最终交付图。
2. 图表的第一目标是让客户、法官或律师快速看懂结构；争点结构图无论简单或复杂，一律使用树状图/分层争点图，不使用 Mermaid `mindmap`。
3. 单张图原则上不超过 25 个节点、每个节点不超过 2 行短句；超过时按“事实阶段、主体类别、争点类别、证明对象”拆图。
4. 图表标题必须说明用途，例如“图1：3亿元投资及资金外流路径图”，不能只写“思维导图”。
5. 每张图后必须有2-3句说明，指出图中主线、关键金额/日期、需要用户关注的节点。

## 第三部分：工作流程

### 步骤1：读取前置文档与系统记录

1. 从当前事项系统记录读取客户信息、案由、当事人信息
2. 读取案件沟通记录（获取客户信息、案件事实、证据初步清单）
3. 读取法律服务建议书（获取请求权路径、构成要件分析、争点清单、证据评估）

### 步骤2：分析法律关系结构

**分析要素：**
- **当事人识别**：原告、被告、第三人、担保人、利害关系人等
- **法律关系类型**：合同关系、侵权关系、担保关系、不当得利、无因管理等
- **权利义务流向**：谁向谁主张什么权利、承担什么义务
- **关键要素标注**：金额、期限、违约事实、损害后果等

**输出：** 法律关系结构分析表

### 步骤3：提取关键时间节点

**时间要素：**
- 合同签订日
- 履行期限/违约发生日
- 催告/协商日
- 重要沟通节点
- 起诉准备日
- 其他影响诉讼时效的日期

**输出：** 关键时间节点列表

### 步骤4：构建争点树结构

基于法律服务建议书中的争点清单：

1. **识别主争点**：案件核心争议
2. **分解子争点**：
   - 事实争议焦点
   - 法律争议焦点
   - 证据争议焦点
3. **关联要件事实**：每个子争点对应的要件事实
4. **关联证据**：每个要件事实的证据支撑
5. **标注状态**：
   - 共识要件（✅绿色）
   - 争议要件（🔴红色）
   - 待确认要件（🟡黄色）
   - 证据状态：✅充分 / ⚠️不足 / ❌缺失

**输出：** 争点清单（树形结构）

### 步骤5：构建要件分析结构

基于法律服务建议书中的请求权基础分析：

1. **确定请求权**：从法律服务建议书获取推荐的请求权路径
2. **列出构成要件**：每个请求权的全部构成要件
3. **关联要件事实**：每个要件对应的具体事实
4. **关联证据**：支撑每个要件事实的证据
5. **标注状态**：✅满足 / ⚠️待证 / ❌不满足

**输出：** 要件分析表

### 步骤6：构建攻防对抗结构

1. **整理我方主张**：基于请求权基础
2. **预测对方抗辩**：基于法律服务建议书中的抗辩权预判
3. **关联争议焦点**：每对攻防对应的争议焦点编号

**输出：** 攻防对照表

### 步骤7：构建证据链图谱

1. **按证明对象分组**：
   - 合同成立
   - 付款/履行事实
   - 违约事实
   - 损害事实
   - 因果关系
   
2. **标注每个证据节点**：
   - 证据名称
   - 证据类型
   - 原件状态
   - 三性评价

3. **标注证据关系**：
   - 印证关系（实线箭头）
   - 补强关系（虚线箭头）
   - 矛盾关系（双向虚线）

**输出：** 证据链关系表

### 步骤8：生成并渲染图片化图表

调用 **mermaid-diagrams** Skill，根据分析结果生成以下图表：

**全套案件图表类型：**
1. 法律关系图（flowchart LR）
2. 法律事实时间图（timeline）
3. 金额流向图（flowchart LR/TB）
4. 争点结构图（树状图/分层争点图）
5. 要件分析图（graph TD）
6. 攻防对抗图（优先泳道式左右对照）
7. 证据链图谱（graph TD）

用户要求“案件可视化图表”“案件思维导图”“案件全套图表”时，必须默认输出上述全套7张图；只有用户明确要求“只要其中某一张图”时，才可缩减范围。

**法官版图表（不含策略分析）：**
1. 法律关系图
2. 法律事实时间图
3. 金额流向图（如案件涉及款项）
4. 争点结构图（去除"对我方有利/不利"等标注）

**律师版图表（全部图表+策略标注）：**
- 全套7种图表
- 包含策略标注和风险提示
- 薄弱环节用红色高亮
- 证据缺失处标注补强建议

## 第四部分：排版与嵌入规范

**Mermaid图表处理原则：**
- 每个图表同时导出 `.mmd` 源文件和 `.png` 图片；复杂图必要时同时导出 `.svg`
- 正式交付稿以图片为主；`.mmd` 仅作为内部可编辑源文件，不作为正式交付稿
- 图表编号：图1、图2...（全套连续编号）
- 生成图片后必须检查是否拥挤、断字、文字重叠、方向不清；不合格时必须拆图或更换图型后重新渲染

> 📖 详细排版格式和嵌入规则参见 [references/图表排版规范.md](图表排版规范.md)

## 第五部分：预览检查点与版本确认

### 5.1 预览稿/工作稿

首次生成时只形成预览稿/工作稿，不得直接称为正式交付稿。

**保存位置：**
- 预览图片：`<当前事项业务文件路径>/<交付目录>/诉讼可视化/预览稿/`
- `.mmd` 源文件和检查记录：当前事项系统记录区或业务文件区的 `诉讼可视化/工作稿/`

**预览稿必须包含：**
- 全套7张图的 `.png` 预览图
- 对应 `.mmd` 源文件
- `图表出稿前审查报告.md`
- `结构体检报告.md`

### 5.2 图表出稿前审查

预览稿生成后必须先审查，审查项目包括：

1. 来源一致性：图中人物、日期、金额、案号、证据名称是否来自已读取材料或明确标注为待确认。
2. 结构完整性：法律关系、事实时间轴、金额流向、争点、要件、攻防、证据链是否覆盖用户要求的案件核心。
3. 可读性：是否存在文字重叠、节点拥挤、断字、箭头方向不清、颜色含义不明或单图信息过载。
4. 版本边界：律师内部版可保留策略标注；客户/法院客观版必须删除内部策略、主观评价和“对我方有利/不利”等标注。
5. 交付路径：正式图片目录是否位于当前事项业务文件区，不得位于 `.cache`、`/tmp`、下载目录或系统记录区。

### 5.3 律师确认选项

出示预览稿/工作稿时，必须同步提示律师选择：

- A 全部确认，制作正式图片交付稿
- B 修改指定图表
- C 仅保留律师内部工作稿
- D 生成客户/法院客观版，删除策略标注

律师确认结果必须写入 `律师确认记录.md`。未取得确认时，不得制作正式图片交付稿。

### 5.4 版本类型

**客户/法院客观版图片包：**
- 纯图表 + 客观图题
- 不含策略分析、主观评价或"对我方有利/不利"标注
- 默认包含法律关系图、法律事实时间图、金额流向图、争点结构图；用户要求全套时可输出客观化全套7图

**律师内部版图片包：**
- 默认包含全套7种图表
- 可包含策略标注、风险提示和证据补强建议
- 薄弱环节可用红色高亮
- 文件名和目录必须标注 `律师内部版` 或 `工作稿`

## 第六部分：本地图片交付包

### 6.1 正式图片交付流程

1. 读取 `当前事项.md`，确认当前事项业务文件路径和系统记录路径。
2. 在当前事项业务文件区创建或使用既有交付目录：`<当前事项业务文件路径>/<交付目录>/诉讼可视化/正式图片交付稿/`。
3. 将经确认的预览稿重新渲染为正式 `.png` 图片；复杂图可同步导出 `.svg`。
4. 对正式图片再次执行结构体检，确认图片可打开、文字可读、节点不重叠、文件命名完整、交付路径正确，并更新 `结构体检报告.md`。
5. 将 `.mmd` 源文件、审查报告、确认记录、结构体检报告保存至工作稿或系统记录目录；正式图片交付目录只放正式图片和必要说明清单。
6. 更新当前事项系统记录中的输出位置、版本、确认状态和完成时间。

## 第七部分：系统记录同步逻辑

> **强制规则**：当前事项系统记录必须同步；复盘台账仅在用户确认或总控规则要求时更新。

### 7.1 同步位置

当前事项系统记录；复盘台账按需确认后同步。

### 7.2 同步字段

生成可视化图片交付包后，更新以下字段：

| 字段名 | 回写内容 |
|-------|---------|
| 案件状态 | "已可视化" |
| 可视化预览稿路径 | 预览稿/工作稿保存位置 |
| 可视化正式图片交付路径 | 正式图片交付稿保存位置 |
| 可视化版本 | 客户/法院客观版、律师内部版或双版本 |
| 律师确认状态 | 已确认、需修改、仅保留工作稿或待确认 |
| 图表出稿前审查报告路径 | `图表出稿前审查报告.md` 路径 |
| 结构体检报告路径 | `结构体检报告.md` 路径 |
| 可视化完成时间 | 当前日期时间 |

### 7.3 字段维护

当前事项系统记录中没有上述字段时，在系统记录中补充；复盘台账不静默扩展字段。

### 7.4 同步异常处理

- 如当前事项系统记录不存在，先提示建立或切换当前事项
- 如复盘台账需要更新但未经确认，先提示用户确认
- 同步失败时写入当前事项的缺口/待办记录并提示用户

- **当前事项系统记录**：同步预览稿路径、正式图片交付路径、律师确认状态、审查报告和结构体检报告路径

- **庭审准备Skill**：争点树、攻防对抗图用于庭审提纲制作

## 第十部分：注意事项

1. **图表准确性**：可视化图表基于已有信息生成，随着案件进展可能需要更新

2. **保密义务**：
   - 客户/法院客观版可对外提交
   - 律师内部版仅限内部使用，请勿对外提供

3. **客户沟通**：可视化图表是与客户沟通的有效工具，建议使用律师版与客户讨论策略

4. **庭审使用**：法官版图表可在庭审中使用，帮助法官理解复杂案情

5. **Mermaid限制**：部分复杂图表可能需要手动调整布局以达到最佳展示效果

6. **前序Skill依赖**：本Skill依赖初步法律分析Skill的输出，建议先完成初步法律分析再执行可视化
