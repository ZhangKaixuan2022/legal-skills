# litigation-analysis

**版本**: 2.0.0
**最后更新**: 2026-03-25
**说明**: 诉讼分析工具，支持起诉状、判决书和庭审笔录全流程分析，整合判决书九层拆解法、上诉/再审可行性评估、败诉原因分析与补救策略建议

## 功能概述

本技能支持三种诉讼文档分析场景：

### 场景一：起诉状与证据分析（诉讼起点）

针对案件初期的起诉状和证据材料，进行快速评估和策略制定：

- **内部版**：诉讼请求分析、事实主张梳理、证据评估、风险识别
- **客户版**：案件情况说明、预期风险、补充证据建议

**核心目标**：快速评估案件基础，识别风险和证据缺口，制定初步策略

### 场景二：判决书分析（核心功能）

针对上级律师或客户发来的判决书（一审/二审），运用**判决书分析九层拆解法**智能分析案件情况，生成三层递进输出：

- **内部版**：面向主办律师，全面深入，专业论证，快速掌握核心要点
- **研究版**：面向律师/研究人员，基于内部版识别法律问题，生成研究课题
- **客户版**：面向非法律专业人士，基于内部版简化呈现，通俗易懂

**核心分析框架：**
- 判决书九层拆解法（基于要件审判九步法）
- 上诉可行性评估矩阵（事实/法律/程序三维度）
- 再审事由检索（对照民诉法第207条）
- 败诉原因分析（证据/法律适用/程序/策略四维度）
- 补救策略建议（路径决策与时间节点提醒）

**核心目标**：理解判决发生了什么，决定是否上诉/再审，帮助赢得客户/案子

### 场景三：庭审笔录分析

针对庭审结束后的笔录文字稿（可能来自录音转写、人工记录或法院系统导出），生成结构化的庭审复盘分析报告：

- **争议焦点归纳**：法庭归纳的核心争议焦点
- **证据质证情况**：双方证据及质证意见
- **法官提问分析**：法官关键提问及可能透露的倾向
- **时限事项整理**：法官要求与时限安排
- **策略建议**：下一步诉讼策略和行动计划

**核心目标**：庭审复盘、团队沟通、策略调整、时限管理

### 与民事一审诉讼Agent的协作说明

- **标准流程调用**：在标准流程中，本工具在判决书送达后由主Agent自动调用（场景二：判决书分析）
- **场景一扩展**：起诉状与证据分析可在起诉状生成后，由律师选择性手动调用
- **场景三扩展**：庭审笔录分析可在庭审结束后，由律师选择性手动调用

## 核心定位

### 主要处理对象

**判决书**（一审判决、二审判决）- 结构化深度抽取
- 这是本技能的核心处理对象
- 通常 100-500 行，信息密集
- 需要完整的事实认定链条、法律逻辑分析、判决结果解读

### 辅助参考文件

当客户或上级律师同时提供其他文件时，可作为辅助参考：

- **起诉状**：了解原告主张和诉讼请求
- **上诉状**：了解上诉理由和新证据
- **答辩状**：了解对方抗辩和反驳证据
- **庭审笔录**：了解庭审过程和质证情况（详见 [庭审笔录处理](hearing-transcript-extraction.md)）

> 注意：起诉状、上诉状、答辩状等不是主要处理对象，仅作为辅助参考，帮助理解判决书内容。庭审笔录可独立处理生成庭审复盘分析报告。

## 核心特性

- **文件夹批量处理**：支持读取用户指定的案件材料文件夹，自动识别并处理所有文件
- **多格式文件支持**：支持 PDF、DOC/DOCX、Excel、Markdown、TXT 等多种格式，自动调用对应 Skill 读取
- **扫描件 OCR 识别**：对扫描件 PDF 自动进行 OCR 识别，确保内容完整提取
- **双场景支持**：判决书分析（核心）+ 庭审笔录分析
- **辅助参考整合**：可同时处理起诉状、上诉状、答辩状等辅助文件
- **结构化深度抽取**：针对判决书/庭审笔录的规范结构进行完整抽取
- **判决书九层拆解法**：基于要件审判九步法，对判决书进行系统性深度分析
- **上诉/再审决策支持**：上诉可行性评估矩阵、再审事由检索、败诉原因分析
- **补救策略建议**：基于分析结果给出具体补救路径和时间节点提醒
- **当前事项系统记录整合**：与复盘台账衔接，读取前序材料、按确认同步分析结果
- **递进输出体系**：判决书三层输出（内部版→研究版→客户版），庭审笔录单层输出（复盘分析报告）
- **决策支持导向**：为上诉/再审决策或庭审策略调整提供专业支持

## 判决书深度分析框架

基于《要件审判九步法》第八、九步逻辑，对判决书进行九层递进式拆解分析：

### 判决书分析九层拆解法

**第一层：提取诉讼请求**
- 原告请求了什么？
- 被告答辩了什么？
- 是否存在反诉？

**第二层：识别法院确定的请求权基础**
- 法院适用了哪条法律？
- 是否与原告主张的请求权基础一致？
- 法院是否进行了释明或变更？

**第三层：分解法院认定的构成要件**
- 法院认为该请求权有哪些构成要件？
- 是否存在隐含要件或非完全性条文补充？

**第四层：逐项分析要件事实认定**
- 每个要件，法院认定的事实是什么？
- 认定依据是什么证据？
- 是否存在推定或自认？

**第五层：证据采信分析**
- 法院采信了哪些证据？排除了哪些？
- 采信/排除的理由是什么？
- 证据"三性"（真实性、合法性、关联性）的判断是否合理？
- 举证责任分配是否正确？

**第六层：法律适用分析**
- 法条引用是否准确？
- 司法解释适用是否恰当？
- 法律推理逻辑是否成立？
- 是否存在法律适用遗漏？

**第七层：要件归入分析**
- 事实→要件的归入是否正确？
- 是否有遗漏的要件？
- 是否有错误归入？
- 归入逻辑是否自洽？

**第八层：裁判结论分析**
- 结论是否与论证过程一致？
- 判赔金额/责任比例的计算是否合理？
- 诉讼费用分担是否正确？

**第九层：程序合法性审查**
- 审判程序是否合法？
- 是否存在程序瑕疵？
- 当事人诉讼权利是否得到保障？

### 每层分析输出格式

| 分析层次 | 法院认定内容 | 是否存在错误 | 错误类型 | 错误严重程度 |
|---------|------------|------------|---------|------------|
| 第X层 | [具体认定] | 是/否/存疑 | 事实认定错误/法律适用错误/程序违法 | 重大/一般/轻微 |

## 上诉可行性评估矩阵

从事实认定、法律适用、程序问题三个维度构建评估矩阵，为上诉决策提供量化支持。

### 上诉可行性评估表

| 评估维度 | 具体问题 | 一审认定 | 我方主张 | 错误程度 | 改判可能性 |
|---------|---------|---------|---------|---------|----------|
| **事实认定** | | | | | |
| 关键事实A | [法院认定内容] | [我方主张] | 重大/一般/无 | 高/中/低 |
| 关键事实B | ... | ... | ... | ... |
| **法律适用** | | | | | |
| 法条适用X | [法院适用] | [应当适用] | 重大/一般/无 | 高/中/低 |
| 法条适用Y | ... | ... | ... | ... |
| **程序问题** | | | | | |
| 程序事项M | [一审做法] | [应当做法] | 重大/一般/无 | 高/中/低 |

### 上诉胜率综合评估

**评估公式：**
- 事实认定错误数量及严重程度 × 权重40%
- 法律适用错误数量及严重程度 × 权重40%
- 程序违法问题 × 权重20%

**综合评估结论：**

| 评估等级 | 判断标准 | 建议 |
|---------|---------|-----|
| 建议上诉 | 存在重大事实认定错误或法律适用错误，改判可能性较高 | 积极准备上诉材料，争取改判 |
| 审慎上诉 | 存在一般性错误，但改判可能性中等 | 权衡成本收益，与客户充分沟通后决策 |
| 不建议上诉 | 错误轻微或无明显错误，改判可能性低 | 建议接受判决或寻求其他救济途径 |

## 使用方式

### 方式一：文件夹批量处理（推荐）

支持读取用户指定的案件材料文件夹，自动识别并处理所有文件：

```
/litigation-analysis @案件材料文件夹路径/
```

**文件夹处理流程**：
1. 使用 `find` 命令扫描文件夹内所有文件
2. 根据文件扩展名自动选择读取方式：
   - **.pdf** → 调用 `/pdf` Skill 读取（自动识别扫描件并 OCR）
   - **.doc/.docx** → 调用 `/docx read` Skill 读取
   - **.xlsx/.xls** → 调用 `/xlsx read` Skill 读取
   - **.md/.txt** → 直接读取文本内容
   - **图片格式**（.jpg/.png/.jpeg）→ OCR 识别（如需要）
3. 自动识别判决书、起诉状、答辩状等文件类型
4. 生成材料清单，包含文件名称、类型、关键内容摘要

### 方式二：基于单个文件


## 文件格式支持

| 格式 | 处理方式 | 说明 |
|------|----------|------|
| **.pdf** | 调用 `/pdf` Skill | 自动检测扫描件，强制 OCR 识别 |
| **.doc/.docx** | 调用 `/docx read` Skill | Word 文档专业读取 |
| **.xlsx/.xls** | 调用 `/xlsx read` Skill | Excel 表格数据提取 |
| **.md/.txt** | 直接读取 | 文本文件直接处理 |
| **图片** | OCR 识别 | 如有需要可进行 OCR |

**重要说明**：
- 必须对扫描件 PDF 进行 OCR 识别，不得忽略
- 读取后的内容将用于后续案件分析
- 所有文件内容将整合到统一的材料清单中

**OCR识别结果质量评估（三级处理，严格执行）**：
- 【可用级】识别结果整体连贯、文字通顺，仅个别字符存疑 → 正常输出，存疑处标注`[OCR识别不确定]`
- 【部分可用级】识别结果部分段落可读、部分段落乱码或缺失 → 输出可读部分，不可读部分标注`[OCR识别失败：该段落无法可靠识别]`，并向用户说明哪些页面/段落无法识别，建议提供清晰扫描件或文字版PDF
- 【不可用级】识别结果整体呈现大面积乱码、字符碎片、语义不连贯，无法支撑准确阅读 → **禁止输出OCR结果**，直接告知用户："该文件为扫描件，OCR识别准确率不足以支撑可靠读取，建议：(1)提供更清晰的扫描件；(2)提供可复制文本的PDF版本；(3)手动输入关键内容"

## Wiki 知识库查阅规则

当处理的案件涉及具体罪名或法律领域时，必须查阅 Wiki 知识库：

1. **检查知识库**：检查 `~/wiki/wiki/` 下是否有对应知识页（刑事：`books/criminal/{罪名}.md`，民事：`books/civil/{领域}/{具体问题}.md`）
2. **读取知识卡片**：若有对应页面，读取页面顶部 `<!-- CARD_START -->` 到 `<!-- CARD_END -->` 之间的知识卡片（约800字浓缩摘要）
3. **按需深入**：卡片信息足够则直接引用；需要深入了解具体辩护策略、案例细节、司法解释原文时，读取全文对应板块
4. **融入输出**：将 Wiki 知识自然融入当前工作输出，引用时标注来源页面路径
5. **案例知识页**：同时检查 `~/wiki/wiki/cases/` 下是否有相关判例知识页可供参考
