# 初步法律分析Skill

## 功能概述

运用"六来源体系"和"请求权基础6步法与要件审判九步法融合框架"，对案件进行系统性法律分析，包括请求权路径识别、构成要件分析、证据评估、抗辩权预测、类案检索和诉讼方案设计，最终生成标准化的法律服务建议书。按总控当前事项记录和复盘台账衔接。

## 触发条件

当用户发送以下格式时自动触发：
- 【客户编号】+ "初步法律分析"
- 【客户编号】+ "法律分析"
- 【客户编号】+ "请求权分析"
- 【客户编号】+ "分析案件"

2. 读取该客户已有信息
3. 获取"沟通记录文件路径"字段值
4. 如有前序沟通记录，读取其中的案件信息

### 1.2 案件沟通记录读取

**默认路径：**
```
<当前事项业务文件路径>/01-委托阶段/案件沟通记录/
```

**或从当前事项系统记录或复盘台账「沟通记录文件路径」字段获取**

### 1.3 证据材料读取

**默认路径：**
```
<当前事项业务文件路径>/01-委托阶段/证据材料/
```

**读取规则**：
1. 扫描证据材料文件夹中的所有文件
2. **逐份证据完整读取全部内容**（禁止摘要、禁止仅提取关键信息、禁止跳过任何页面）
3. 支持格式：Word（.doc/.docx）、PDF（.pdf）、Excel（.xls/.xlsx）、图片（.jpg/.png）

**PDF文件处理规范**：
- **可复制文本PDF**：直接全文读取，逐页提取完整文字内容
- **扫描件PDF（图片型PDF）**：使用pdf Skill进行OCR文字识别，读取全部内容
- **OCR识别结果质量评估（三级处理，严格执行）**：
  - 【可用级】识别结果整体连贯、文字通顺，仅个别字符存疑 → 正常输出，存疑处标注`[OCR识别不确定]`
  - 【部分可用级】识别结果部分段落可读、部分段落乱码或缺失 → 输出可读部分，不可读部分标注`[OCR识别失败：该段落无法可靠识别]`，并向用户说明哪些页面/段落无法识别，建议提供清晰扫描件或文字版PDF
  - 【不可用级】识别结果整体呈现大面积乱码、字符碎片、语义不连贯，无法支撑准确阅读 → **禁止输出OCR结果**，直接告知用户："该文件为扫描件，OCR识别准确率不足以支撑可靠读取，建议：(1)提供更清晰的扫描件；(2)提供可复制文本的PDF版本；(3)手动输入关键内容"
- **禁止部分读取**：无论PDF页数多少，必须逐页完整读取，不得仅读取前几页或摘要

在请求权确定阶段，使用"六来源体系"作为标准分析框架，系统性梳理案件的主体、管辖、权利、救济、成就、内容六个关键维度。

> 📖 参考标准参见 [references/六来源体系表.md](六来源体系表.md)

### 2.2 请求权基础6步法与要件审判九步法融合框架

**融合逻辑：** 6步法作为宏观框架，九步法提供微观执行细节。融合后形成8步分析流程，贯穿从固定权利请求到制订实施方案的完整过程。

> 📖 参考标准参见 [references/6步法与9步法融合框架.md](6步法与9步法融合框架.md)

## 第三部分：分析工作流（8步融合流程）

### 步骤1：固定权利请求

**操作指令：**
1. 从案件沟通记录中提取：
   - 客户基本信息
   - 案件事实经过（按时间轴整理）
   - 客户明确表达的诉求
   - 现有证据材料清单

2. 固定权利请求：
   - 识别请求权类型（给付之诉/确认之诉/变更之诉）
   - 明确请求权内容（要钱/要物/要行为/要确认）
   - 确定请求权主体和对象

3. **运用六来源体系分析：**
   - 主体来源分析（原被告适格性）
   - 管辖来源分析（管辖法院确定）
   - 权利来源初步识别
   - 救济来源梳理
   - 成就来源检查
   - 内容来源明确

**输出要求：**
- 权利请求固定表
- 六来源体系分析表

**智能追问触发条件：**
> 当事人信息不完整时："案件涉及的对方当事人，您能否补充：A. 完整名称/姓名；B. 联系方式；C. 住所地址？这些信息对确定管辖法院和送达都很重要。"

---

### 步骤2：确定请求权基础规范

**操作指令：**
1. 基于案件事实和六来源分析，识别可能的请求权基础（合同请求权、侵权请求权、不当得利返还请求权、无因管理请求权、物权请求权、人格权请求权、其他特别法请求权）

> 📖 参考标准参见 [references/请求权基础识别指南.md](请求权基础识别指南.md)

2. 对每个请求权路径：
   - 引用具体法律条文
   - 分析与案件事实的契合度
   - 进行初步可行性判断

**联网检索：**
- 检索相关法律法规（国家法律法规数据库等公开渠道）
- 检索局限需在后续意见书中注明

**输出要求：** 请求权基础分析表

**智能追问触发条件：**
> 存在多个请求权路径时："本案存在多个可行的请求权路径，各有优劣。您更倾向于：A. 追求胜诉概率最大化；B. 追求执行可能性最大化；C. 追求时间成本最小化？"

---

### 步骤3：确定抗辩权基础规范

**操作指令：**
1. 系统检索对方可能的抗辩权（权利未发生抗辩、权利已消灭抗辩、权利行使受阻抗辩、时效抗辩、过失相抵、损益相抵、免责事由、约定免责、其他抗辩）

> 📖 参考标准参见 [references/常见抗辩权清单.md](常见抗辩权清单.md)

2. 分析每项抗辩：
   - 法律依据
   - 成立条件
   - 对方是否可能具备该抗辩条件

**输出要求：** 抗辩权基础规范表

---

### 步骤4：构成要件分析

**操作指令：**
1. 对每个可行的请求权基础，进行构成要件分解
2. 包含非完全性条文补充（如《民法典》第五百七十七条需补充违约行为类型）
3. 推导隐含要件（如因果关系中的可预见性规则）

**示例：合同违约之诉**
- 要件一：合同成立且生效
- 要件二：存在违约行为
- 要件三：造成损害后果
- 要件四：因果关系
- 要件五：违约方有过错（如适用严格责任则无需）

4. **对照案件事实评估每个要件：**
   - 已有证据支持的要件
   - 证据不足的要件
   - 存在争议的要件

**输出要求：** 构成要件分析表

---

### 步骤5：诉讼主张检索与争点整理

**操作指令：**
1. 梳理我方主张（基于请求权基础）
2. 预测对方主张（基于抗辩权分析）
3. 形成争点清单：
   - **共识要件**：双方无争议、证据充分的要件
   - **争议要件**：可能产生争议、需要重点证明的要件

4. 识别争议焦点类型：
   - 事实争议焦点
   - 法律适用争议焦点
   - 证据争议焦点

**输出要求：** 争点清单表

**智能追问触发条件：**
> 案件事实存在多种解释时："关于【某事实】，存在两种可能的解释：A...；B...。哪种更符合实际情况？"

---

### 步骤6：要件事实证明策略

**操作指令：**
1. 为每个争议要件设计证明方案
2. **证据三性评估：**
   - 真实性：证据来源、形成过程、是否有伪造可能
   - 合法性：取证方式、证据形式是否合法
   - 关联性：与待证事实的关联程度

3. **证据视角分析（每个请求权路径）：**
   - 核心证据类型识别
   - 推定与自认风险标注
   - 证据保全必要性评估

4. **举证责任分配分析框架**

   > 📖 参考标准参见 [references/证明责任分配速查表.md](证明责任分配速查表.md)

   依据法律要件分类说（规范说）进行举证责任分配：

   - **权利发生规范（基本规范）**：主张权利存在的一方对权利发生的法律要件事实承担举证责任
   - **对立规范**：
     - 权利妨碍规范：主张权利未有效成立（如合同无效、行为能力欠缺）
     - 权利消灭规范：主张权利已消灭（如已履行、已抵销、已免除）
     - 权利限制规范：主张权利受限制（如诉讼时效经过、同时履行抗辩权）
   - **举证责任倒置的识别**：法律特别规定的情形（如环境污染、产品责任、医疗损害等）

   法律依据：《最高人民法院关于适用〈中华人民共和国民事诉讼法〉的解释》第90条、第91条

   输出格式：按每个请求权要件列出举证责任归属方

5. **证明标准层次化判断**

   根据待证事实性质适用不同证明标准：

   - **一般标准——高度盖然性**：证据使待证事实的存在比不存在更具有高度可能性（《民诉解释》第108条）
   - **提高标准——排除合理怀疑**：适用于欺诈、胁迫、恶意串通事实的证明，口头遗嘱或赠与事实的证明（《民诉解释》第109条）
   - **降低标准——较大可能性**：适用于程序性事项的证明

   输出：判断本案各待证事实适用的证明标准层级及理由

6. **证据可行性评估与证明风险预判**

   对每个要件事实进行系统性评估：

   - **证据支撑程度评估**：现有证据对要件的支撑程度（充分/不足/缺失）
   - **败诉风险分析**：关键事实真伪不明时的结果意义举证责任归属
   - **证据收集可行性评估**：补充证据的成本、难度、时效性

   输出：证明风险评估表

   | 要件事实 | 现有证据支撑度 | 证明风险等级（高/中/低） | 补充取证建议 |
   |---------|--------------|---------------------|------------|
   |         |              |                     |            |

**输出要求：**
- 证明策略表
- 证据三性评估表

**智能追问触发条件：**
> 证据不足时："【某要件】目前证据不足，您能否补充：A. 书面证据；B. 证人证言；C. 其他？"

---

### 步骤7：事实认定预判

**操作指令：**
对每个争议要件进行事实认定预判，采用三种形态：

| 认定形态 | 说明 | 后续策略 |
|---------|------|---------|
| 支持我方 | 现有证据足以证明该要件成立 | 巩固证据，防止对方攻击 |
| 支持对方 | 现有证据对我方不利 | 补充证据或调整诉讼策略 |
| 证据不足 | 双方证据均不充分 | 加强取证或利用证明责任分配 |

**输出要求：** 事实认定预判表

---

### 步骤8：制订实施方案

**操作指令：**

**8.1 构建请求权路径对比表（风险矩阵）**

当存在多条请求权路径时：

| 对比维度 | 路径A | 路径B | 路径C |
|---------|------|------|------|
| 请求权基础 | | | |
| 构成要件满足度 | | | |
| 证据充分度（高/中/低） | | | |
| 对方可能抗辩点 | | | |
| 胜诉概率评估 | | | |
| 执行可行性 | | | |
| 推荐优先级 | | | |

**8.2 类案检索与分析**

**联网检索：**
1. 检索相似案例：
   - 优先：最高人民法院指导性案例、公报案例
   - 其次：审理法院所在地高级法院、中级法院案例
   - 再次：相似案情案例

2. 分析裁判规则：
   - 归纳裁判要点
   - 分析法官观点
   - 识别类案与本案的异同

**注意：** 基于公开渠道检索，建议后续在付费数据库中补充检索

**8.3 诉讼方案设计**

**方案设计原则：**
- 至少提供两个方案供选择
- 最优方案 + 备选方案
- 各方案需包含优劣势分析

**每个方案包含：**
1. 方案名称
2. 诉讼主体（原告/被告/第三人）
3. 诉讼请求（主请求/备位请求）
4. 管辖法院（法律依据+选择理由）
5. 请求权基础
6. 证据组织策略
7. 优势分析（事实/法律/证据）
8. 风险提示（事实/法律/证据/程序风险）
9. 预期效果（胜诉概率/执行可能性/时间成本）

**输出要求：**
- 请求权路径对比表
- 类案检索报告
- 诉讼方案对比表

**智能追问触发条件：**
> 需确认诉讼目标优先级时："如果要在以下目标中选择，您的优先级是：A. 快速回款；B. 彻底胜诉；C. 最小成本？"

## 第六部分：交互规范

### 6.1 智能追问机制

**追问原则：**
- 每次只追问一个关键问题
- 追问语言简洁专业
- 说明追问的原因（为什么需要这个信息）
- 等待用户回复后再继续

**追问情形汇总：**

| 触发情形 | 追问示例 |

## 第八部分：注意事项

1. **检索局限说明**：必须在建议书中注明检索基于公开渠道，部分资源需登录账号，建议后续补充检索

2. **分析局限性**：初步法律分析基于现有信息和公开渠道检索，随着案件进展和证据补充，分析结论可能需要调整

3. **不构成承诺**：法律服务建议书仅供决策参考，不构成对案件结果的承诺或保证

4. **保密义务**：所有案件信息严格保密，仅用于本案办理

5. **时效性**：法律法规和裁判规则可能更新，建议定期复核

6. **不包含费用预算**：本Skill生成的法律服务建议书不包含费用预算内容，费用相关事宜由委托合同Skill处理

7. **记录同步异常处理**：
   - 如当前事项系统记录不存在，提示建立或切换当前事项
   - 如客户编号未能定位当前事项，新建记录
   - 回写失败时保存备份并提示用户

## Wiki 知识库查阅规则

当处理的案件涉及具体罪名或法律领域时，必须查阅 Wiki 知识库：

1. **检查知识库**：检查 `~/wiki/wiki/` 下是否有对应知识页（刑事：`books/criminal/{罪名}.md`，民事：`books/civil/{领域}/{具体问题}.md`）
2. **读取知识卡片**：若有对应页面，读取页面顶部 `<!-- CARD_START -->` 到 `<!-- CARD_END -->` 之间的知识卡片（约800字浓缩摘要）
3. **按需深入**：卡片信息足够则直接引用；需要深入了解具体辩护策略、案例细节、司法解释原文时，读取全文对应板块
4. **融入输出**：将 Wiki 知识自然融入当前工作输出，引用时标注来源页面路径
5. **案例知识页**：同时检查 `~/wiki/wiki/cases/` 下是否有相关判例知识页可供参考
