# 来源与边界协议

## Reviewer Note

所有正式法律输出前必须有“来源与边界说明”。该说明集中放在正文前，不在正文中散落元说明。

如果全部清楚，可压缩成一行：

```markdown
【来源与边界说明】已读取全部指定材料；法规已联网校验；无明显材料缺口；供律师复核后使用。
```

如存在未读、未验证、存疑或需要判断的事项，使用完整格式。

## 来源标签

重要结论必须使用以下标签之一：

```text
[文件原文：文件名/页码或条款]
[用户口述：日期]
[北大法宝MCP：法规名/Gid/日期]
[法规联网校验：法规名/条号/来源/日期]
[案例检索：案号/来源/日期]
[Wiki：页面路径]
[网页搜索：URL/日期]
[模型推理-基于已读材料]
[模型知识-未验证]
[未验证]
[需用户判断]
[材料缺口]
```

标签描述实际发生的来源行为，不能虚标。

## 验证规则

- 法规现行有效状态默认优先通过北大法宝 MCP/API 或官方源确认。
- 北大法宝 MCP/API 已跑通，可用于核验法规名称、Gid、发文字号、发布机关、发布日期、实施日期、时效状态和法宝链接。
- 北大法宝列表接口未返回条文原文时，不得标注为条文原文已核验。
- 案例裁判规则必须读到原文或可靠摘要后才能引用。
- 网页搜索结果必须标明 URL 和日期。
- Wiki 不能替代现行法校验。
- 模型记忆只能作为待验证线索。

## 冲突处理

当文件、用户口述、网页检索、模型知识之间冲突时：

1. 暴露冲突。
2. 标明各自来源。
3. 不静默选择一方。
4. 需要用户或进一步检索确认时，写入缺口并提示用户。

## 材料不足

材料不足时只能：

1. 停下来提示用户补材料。
2. 继续但明确标注为“基于现有材料的初步意见”。
3. 写入缺口归档，后续补充。

不得悄悄脑补事实、金额、日期、案号、法律依据或案例结论。

## 缺口提示

缺口不只归档，还必须在当前回复中提示：

```markdown
【需要你判断/补充】
- 缺口：
- 影响：
- 建议补充：
- 如继续：只能形成基于现有材料的初步意见。
```

## 客户/外部交付

正式交付文书正文应保持专业干净。来源与边界说明放在正文前或单独说明中；必要来源可用脚注、尾注或括号标签。
