# 审理报告Skill

## 功能概述

从已有案件材料中提取要素，生成法院内部审理报告Word初稿。审理报告是承办人向审判组织（合议庭/审判委员会）汇报案件情况的内部分析论证文书，语体为客观分析、论证评价语体，非正式裁判宣告语体。

适用场景包括：
- **合议庭讨论前的案件汇报**：整理案件要素，形成内部讨论材料
- **审委会汇报材料**：为审判委员会讨论提供案件分析报告
- **案件审结前的内部总结**：梳理争议焦点、证据认定、处理意见

支持与复盘台账衔接，读取当前事项系统记录并按需同步审理报告路径。

## 排版格式规范

审理报告排版与判决书基本一致，使用仿宋三号/行距28磅，但存在以下差异：
- 无法院名称（内部文书）
- 标题格式不同：案件名称分两行排列
- 一级/二级标题使用仿宋三号不加粗（判决书正文统一仿宋）
- 无上诉告知、审判人员签名等尾部信息

> 详细排版标准参见 [references/审理报告排版规范.md](审理报告排版规范.md)

## 工作流程

### 步骤1：读取系统记录/复盘台账

**数据来源**：复盘台账（按法律工作总控确认后读取/更新）

**读取字段**：
| 字段 | 用途 |
|-----|------|
| 客户称呼 | 识别客户 |
| 案件文件夹路径 | 定位案件材料目录 |
| 案号 | 填写案号 |
| 案由 | 填写案由及标题中的纠纷类型 |
| 当事人信息 | 填写原告/被告/第三人信息 |
| 诉讼代理人信息 | 填写代理人信息 |
| 立案日期 | 填写案件由来 |

**操作指令**：
1. 根据当前事项系统记录定位事项；必要时参考复盘台账
2. 读取上述字段
3. 确认案件文件夹路径存在且可访问

### 步骤2：扫描案件文件夹，识别已有文档

**案件文件夹扫描路径**：
```
<当前事项业务文件路径>/
├── 01-委托阶段/
│   ├── 案件沟通记录/
│   └── 法律服务建议书/
├── 02-诉前准备/
│   ├── 起诉状/
│   ├── 答辩状/
│   ├── 法规案例汇编/
│   └── 诉讼可视化/
├── 03-诉讼阶段/
│   ├── 代理词/
│   ├── 质证意见/
│   └── 庭审纪要/
├── 04-庭审阶段/
│   ├── 庭审纪要/
│   ├── 庭审复盘/
│   └── 案件讨论与提纲/
└── 05-裁判阶段/
    └── 判决书/
```

**识别以下文档（如存在）**：
| 文档类型 | 典型文件名 | 提取要素 |
|---------|-----------|---------|
| 起诉状 | 民事起诉状_*.docx/pdf | 原告诉请、事实与理由、当事人信息、证据清单 |
| 答辩状 | 民事答辩状_*.docx/pdf | 被告答辩意见、证据清单 |
| 法律服务建议书 | 法律服务建议书_*.docx/pdf | 法律关系定性、请求权基础、法律依据 |
| 初步法律分析 | 法律分析_*.docx/pdf | 争议焦点、法律适用意见、处理建议、类案检索 |
| 庭审纪要 | 庭审纪要_*.docx/pdf | 争议焦点、双方主张、证据质证情况、事实认定 |
| 庭审复盘 | 庭审复盘_*.docx/pdf | 裁判思路、处理意见 |
| 案件讨论与提纲 | 案件讨论_*.docx/pdf | 裁判思路、处理建议 |
| 法规案例汇编 | 法规案例汇编_*.docx/pdf | 引用法条、类案信息 |

**读取规范**：
- 逐份文档完整读取全部内容
- PDF扫描件使用OCR识别，精度不足时标注 `[OCR识别不确定]`
- 禁止部分读取或仅提取摘要

### 步骤3：按【标题/案号 + 七大区块 + 附件】逐一提取要素，标记缺失项

#### 区块1：标题

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 原告姓名/名称 | 系统记录/复盘台账 / 起诉状 | 是 |
| 被告姓名/名称 | 系统记录/复盘台账 / 起诉状 | 是 |
| 案由 | 系统记录/复盘台账 / 起诉状 | 是 |

**标题格式**：
```
关于原告XX、XX与被告XX、XX及第三人XX XX纠纷一案

审 理 报 告
```

**格式要点**：
- 第一行以"关于"开头，列出全部当事人（含第三人），多个当事人用顿号分隔
- 第二行"审 理 报 告"五字之间各空一格，宋体二号加粗居中
- 案号位于标题下方，右对齐

**缺失处理**：当事人信息或案由缺失时标记为【待补充】，进入步骤4向用户确认。

#### 区块2：案号

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 案号 | 系统记录/复盘台账 | 是 |

#### 区块3：案件由来和审理经过

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 立案日期 | 系统记录 / 起诉状 | 是 |
| 适用程序 | 固定为"普通程序"（或依实际） | 是 |
| 案由 | 系统记录 / 起诉状 | 是 |
| 是否公开开庭 | 庭审纪要 / 默认"公开" | 是 |
| 到庭当事人 | 庭审纪要 | 是 |
| 未到庭说明 | 庭审纪要 | 否 |
| 审理进程 | 庭审纪要 | 有则填 |

#### 区块4：当事人基本情况

| 要素 | 自然人 | 法人/非法人组织 | 数据来源 |
|-----|-------|----------------|---------|
| 原告姓名/名称 | 必填 | 必填 | 系统记录 / 起诉状 |
| 性别 | 必填 | 不适用 | 系统记录 / 起诉状 |
| 出生日期 | 必填 | 不适用 | 系统记录 / 起诉状 |
| 民族 | 有则填 | 不适用 | 系统记录 / 起诉状 |
| 职业/工作单位 | 有则填 | 不适用 | 系统记录 / 起诉状 |
| 住址/住所地 | 必填 | 必填 | 系统记录 / 起诉状 |
| 法定代表人/主要负责人 | 不适用 | 必填 | 系统记录 / 起诉状 |
| 委托诉讼代理人 | 有则填 | 有则填 | 系统记录 / 起诉状 |
| 被告信息 | 同上原告字段 | 同上原告字段 | 系统记录 / 起诉状 / 答辩状 |
| 第三人信息 | 同上原告字段 | 同上原告字段 | 如有则填 |

**缺失处理**：当事人基本信息缺失时标记为【待补充】，进入步骤4向用户确认。

#### 区块5：当事人诉辩主张和举证情况

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 原告诉讼请求 | 起诉状 | 是 |
| 原告事实与理由 | 起诉状 | 是 |
| 原告证据清单及证明目的 | 起诉状 / 庭审纪要 | 是 |
| 被告答辩意见 | 答辩状 / 庭审纪要 | 是 |
| 被告证据清单及证明目的 | 答辩状 / 庭审纪要 | 是 |
| 第三人陈述意见 | 庭审纪要 / 如有第三人 | 否 |

**动态标题规则**：
区块三标题不是固定的，应根据案件当事人情况动态生成。标题格式为：
```
原告的诉请和事实与理由以及【被告名称】答辩、举证、质证情况
```

- 如存在多个被告且答辩意见不同，标题中应分别列出各被告
- 内部结构固定包含：
  - （一）原告诉请和事实与理由 + 原告举证情况
  - （二）被告答辩 + 被告举证情况
  - （三）第三人述称（如有第三人）

#### 区块6：对证据的分析认证、对案件事实的认定

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 证据交换情况 | 庭审纪要 | 是 |
| 质证情况 | 庭审纪要 | 是 |
| 各方证据分析认证（真实性/合法性/关联性） | 庭审纪要 / 法律分析 | 是 |
| 事实认定要点 | 庭审纪要 / 法律分析 | 是 |

**写作格式**：
```
（一）证据分析认证
逐一分析各方证据的真实性、合法性、关联性：
1. 【证据1的分析认证意见】；
2. 【证据2的分析认证意见】；
……

（二）案件事实认定
经审理查明：
1. 【事实认定要点1】；
2. 【事实认定要点2】；
……
```

**缺失处理**：现有材料仅包含部分证据/质证记录，本报告中的证据分析和事实认定仅基于已有材料。如需完整证据分析，请补充庭审纪要或相关证据材料。

#### 区块7：对争议焦点的归纳和评价

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 争议焦点归纳 | 法律分析 / 庭审纪要 | 是 |
| 各争议焦点分析评价 | 法律分析 / 庭审纪要 | 是 |

**写作格式**：
```
争议焦点一：【争议焦点内容】
分析评价：【对该争议焦点的法律分析、证据评价、倾向性意见】

争议焦点二：【争议焦点内容】
分析评价：【对该争议焦点的法律分析、证据评价、倾向性意见】
```

**缺失处理**：根据现有材料，我初步归纳了以下争议焦点及分析意见，请确认是否准确，如有遗漏或需要调整，请补充说明。

#### 区块8：需要说明的问题

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 与上级法院沟通情况 | 用户补充 / 案件材料 | 有则填 |
| 程序性问题 | 用户补充 / 案件材料 | 有则填 |
| 社会影响评估 | 用户补充 / 案件材料 | 有则填 |
| 请示事项 | 用户补充 / 案件材料 | 有则填 |

**重要规则**：
- **此区块优先从案件材料中提取**
- 如材料中无相关内容，**AI必须主动提示用户补充**，不得留空或编造
- 提示语："案件材料中未找到需要特别说明的问题（如与上级法院沟通情况、程序性问题等），请问是否有需要在本报告中特别说明的情况？"

#### 区块9：处理意见

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 案件法律性质认定 | 法律分析 | 是 |
| 法律适用分析（含新旧法对照） | 法律分析 / 法规案例汇编 | 有则填 |
| 各项诉讼请求处理建议 | 法律分析 / 庭审复盘 | 是 |
| 风险提示（上诉风险点、程序注意事项） | 法律分析 / 案件讨论 | 有则填 |

**写作格式**：
```
（一）关于案件法律性质的认定
【对案件法律关系的定性分析，含法律适用分析】

（二）关于法律适用的时效性说明
【如涉及新旧法对照，说明应适用的法律及理由】

（三）关于各项诉讼请求的处理建议
1. 【诉讼请求1的处理建议】；
2. 【诉讼请求2的处理建议】；
……

（四）风险提示
【上诉风险点、程序注意事项等】
```

**重要规则**：
- 相关法律法规不再单列成章，在处理意见中引用适用法条即可
- 引用法条必须通过网络检索确认准确性和时效性

#### 附件：类案检索情况及报告

| 要素 | 数据来源 | 是否必填 |
|-----|---------|---------|
| 检索条件 | 法律分析 / 法规案例汇编 | 有则填 |
| 类案检索结果 | 法律分析 / 法规案例汇编 | 有则填 |

**附件格式**：
```

一、检索条件
关键词：【检索关键词】
范围：【检索范围】
时间：【检索时间】
侧重：【检索侧重】

二、类案检索结果

（一）【案例1案号】
审理法院：【法院名称】
裁判日期：【日期】
当事人：【当事人信息】
案件概要：【简要案情】
裁判要旨：【核心裁判观点】
与本案关联性分析：【关联性说明】

（二）【案例2案号】

```

**重要规则**：
- **类案检索从正文区块改为附件形式**

- 如材料中类案信息不足，**AI应提示用户补充**
- 提示语："案件材料中的类案检索信息不够充分，如需补充类案检索情况，请提供相关检索结果。"

#### 4.2 缺失项补充规则

**必须用户确认/提供的要素**：
1. **证据分析/事实认定**：当材料明显不足时，AI仅可给出初步建议，必须请用户确认是否采纳
2. **争议焦点归纳评价**：当材料明显不足时，AI仅可给出初步建议，必须请用户确认是否采纳
3. **需要说明的问题**：与上级法院沟通情况、社会影响评估、程序性问题等（如材料中未涉及）
4. **类案检索情况**：如材料中类案信息不足，需用户提供（作为附件补充）
5. **处理意见**：AI给出的处理意见必须标注为"根据已有材料整理"，最终处理意见须经用户确认

**可选补充项**：
- 第三人信息（如案件有第三人）
- 不公开开庭理由（如不公开审理）
- 人民陪审员信息（如合议庭有人民陪审员）
- 新旧法适用说明（如涉及法律适用时效性问题）

#### 5.3 文档校验

```javascript
const fs = require('fs');

function validateDocx(filePath) {
    if (!fs.existsSync(filePath)) {
        return { success: false, message: `文档生成失败：文件不存在 (${filePath})`, size: 0 };
    }
    const stats = fs.statSync(filePath);
    const minSize = 3 * 1024;
    if (stats.size < minSize) {
        return { success: false, message: `文档可能损坏：文件过小 (${(stats.size/1024).toFixed(2)}KB < ${minSize/1024}KB)`, size: stats.size };
    }
    const buffer = fs.readFileSync(filePath);
    const isValidZip = buffer.length >= 2 && buffer[0] === 0x50 && buffer[1] === 0x4B;
    if (!isValidZip) {
        return { success: false, message: `文档格式异常：不是有效的ZIP/DOCX格式`, size: stats.size };
    }
    return { success: true, message: `文档校验通过，大小: ${(stats.size/1024).toFixed(2)}KB`, size: stats.size };
}
```

校验不通过时，排查并修复后重新生成。

## 写作规则

### 规则1：多原被告具体指代规则

当案件存在多个原告或多个被告时：

- **禁止**直接用"原告"或"被告"笼统替代全部原告或被告
- **例外情况**（以下情况可统称）：
  - 所有原告的诉讼请求完全一致，可统称"原告"
  - 原告的主张针对全部被告，可统称"被告"
  - 所有被告的答辩意见完全一致，可统称"被告"
- **否则必须区分**：使用具体姓名或公司名称指代，明确是哪一个原告或被告

**错误示例**：
> "被告辩称，……"（案件有两个被告，答辩意见不同）

**正确示例**：
> "被告动力源公司辩称，……"
> "被告合大公司辩称，……"

### 规则2：公司缩写一致性规则

- 在审理报告中**首次出现公司全称**时，应标注缩写：
  > "北京动力源科技股份有限公司（以下简称'动力源公司'）"
- **此后全文统一使用该缩写**，不得出现同一公司的不同缩写
- **缩写命名规范**：取公司名称中最具辨识度的关键词 + "公司"
- **AI在提取要素时应自动识别**并建立公司名称→缩写的映射表，确保全文一致

**错误示例**：
> 前文用"动力源公司"，后文用"动力源"或"北京动力源公司"

**正确示例**：
> 全文统一使用"动力源公司"

## 注意事项

1. **语体要求**：审理报告为内部分析论证文书，语气应为客观分析、论证评价语体，非正式裁判宣告语体。避免使用"本院认为""判决如下"等裁判宣告用语，改用"经分析，……""承办人认为，……"等分析论证用语。
2. **处理意见仅供参考**：AI整理的处理意见必须标注为"根据已有材料整理"，最终处理意见须经用户确认
3. **需说明问题不可编造**：与上级法院沟通情况、社会影响评估等敏感信息必须用户提供或材料中有明确记载，AI不得编造
4. **类案检索需审慎**：如材料中无类案信息，应明确提示用户补充，不得虚构类案
5. **法律依据需校验**：引用法条必须通过网络检索确认准确性和时效性
6. **日期格式统一**：使用"××××年××月××日"格式
7. **案号格式**：按（年份）+法院简称+民初+序号
